# 项目优化建议（2026-04）

基于当前代码与结构，建议按“先稳后快”的顺序推进：先做安全与数据一致性，再做性能与可维护性。

## P0（建议优先本周完成）

1. **会话密钥改为必配项（生产环境）**
   - 现状：`APP_SESSION_SECRET` 未配置时会自动生成随机值，重启后全部会话失效。
   - 风险：生产环境中会导致频繁掉线，且难以排查。
   - 建议：在 `APP_ENV=prod` 时强制要求配置 `APP_SESSION_SECRET`，未配置直接拒绝启动并给出明确错误。

2. **补充 CSRF 防护（所有写操作）**
   - 现状：系统大量使用表单 POST/删除接口，但没有看到 CSRF Token 校验机制。
   - 风险：登录态下可能被跨站请求伪造触发误操作。
   - 建议：为所有 POST 表单加入 CSRF token（可用 `starlette-wtf` 或自定义中间件+签名 token）。

3. **金额字段由 `Float` 迁移为 `Decimal/Numeric`**
   - 现状：收费金额、税率等核心字段大量采用 `Float`。
   - 风险：累计计算时会有浮点误差，影响财务对账。
   - 建议：模型改为 `Numeric(18,2)`（金额）与 `Numeric(6,4)`（税率），服务层统一 `Decimal` 计算。

## P1（建议 1~2 周内完成）

4. **提醒任务减少数据库提交频率，增加批量提交策略**
   - 现状：提醒任务每处理一条记录就 `session.commit()` 一次。
   - 影响：记录量大时事务开销高，吞吐下降。
   - 建议：改为分批（如每 50 条）提交；同时记录失败明细，避免一次网络波动影响全部任务结果。

5. **减少 N+1 查询（列表和提醒消息）**
   - 现状：提醒消息读取 `record.company`/`record.pipeline_entry`，列表页也存在关联对象渲染。
   - 影响：数据量增大后 SQL 次数显著增加。
   - 建议：对关键查询加 `selectinload/joinedload`，并开启 SQL 计数日志观察优化效果。

6. **为高频筛选字段补索引**
   - 现状：`fee_records` 按 `company_id`、`payment_status`、`planned_receivable_date` 查询频繁，但模型中缺少对应索引声明。
   - 建议：添加单列索引与组合索引（如 `(payment_status, planned_receivable_date)`）。

7. **为提醒与管理场景增加分页**
   - 现状：多个列表接口直接 `.all()` 全量加载。
   - 影响：数据大时页面响应与内存占用恶化。
   - 建议：统一分页参数（`page`, `page_size`），页面支持“筛选 + 分页 + 排序”。

## P2（建议后续迭代）

8. **把“税率规则/收费类型规则”集中配置**
   - 现状：税率规则写在路由函数里（入廊费=9%，运维费=6%）。
   - 风险：业务变更时需要多处改代码。
   - 建议：迁移到配置表（或常量配置模块），路由层仅做调用。

9. **统一时间与审计字段**
   - 现状：`created_at/updated_at` 存在，但仍有部分实体缺少更完整审计信息（创建人、更新人、来源）。
   - 建议：补操作日志表，关键变更（收费金额、状态、合同）保留审计轨迹。

10. **数据库迁移工具标准化**
    - 现状：通过 `ensure_sqlite_schema()` 运行时补列。
    - 风险：多环境与长期演进时可追溯性不足。
    - 建议：引入 Alembic 管理版本化迁移；保留一次性兼容脚本用于历史库。

## 推荐落地顺序

- 第 1 阶段：P0（安全与资金精度）
- 第 2 阶段：P1（性能与可扩展）
- 第 3 阶段：P2（治理与长期维护）

## 可量化验收指标（建议）

- 登录会话 7 天内稳定率 ≥ 99%（服务重启不强制掉线）
- `fee_records` 列表在 1 万条数据下首屏响应 < 1 秒
- 提醒任务在 500 条记录下执行耗时下降 30% 以上
- 财务抽样对账差异 = 0（使用 Decimal 后）
